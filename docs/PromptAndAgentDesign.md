# Agent 设计记录

## 当前版本

当前版本使用本地规则模拟 AI 运营 Agent 和 NPC Agent，暂未调用外部模型。

原因：

1. 第一版重点验证游戏运营闭环。
2. 避免在未建立安全边界前让模型直接影响游戏数值。
3. 便于后续把规则输出替换为真实 LLM 调用。
4. 第二版重点验证 NPC 记忆、工具读取和规则约束的 Agent 链路。
5. 第四版重点验证运营 Agent 对指标、异常、调参和活动配置草案的结构化输出。
6. 第五版重点验证 Agent 草案进入配置工作台后的校验、人工确认、应用和回滚流程。

## Agent 输入

### 运营 Agent

1. 游戏天数。
2. 事件埋点。
3. 商品配置。
4. 接待人数、成交数、收入、流失次数、转化率。
5. 任务完成率、商品购买转化、NPC 互动率、流失节点和 A/B 测试结果。

### NPC Agent

1. NPC 角色设定。
2. NPC 偏好、厌恶项和预算。
3. 当前库存和商品价格。
4. NPC 历史记忆：好感、到访次数、最近购买、最近反馈、动态情绪。
5. 当前任务目标。

## Agent 输出

### 运营 Agent

1. 运营摘要。
2. 数据证据。
3. 风险判断。
4. 运营建议。
5. 活动配置草案。
6. 异常与原因假设。
7. 行动计划。
8. 数值调参建议。
9. 人工确认清单。

### NPC Agent

1. 商品推荐理由。
2. 角色化对话反馈。
3. 记忆更新建议。
4. 工具调用轨迹。
5. 规则边界说明。

## 安全边界

1. Agent 只能生成建议和配置草案。
2. Agent 不允许直接修改金币、声望、满意度等关键数值。
3. 后续真实模型输出必须通过 schema 校验。
4. 配置进入游戏前必须人工确认。
5. NPC Agent 不能直接决定销售结果；成交必须经过库存、预算、偏好和规则校验。
6. NPC Agent 不能直接任意增加好感；好感变化由命中偏好、替代推荐和失败接待规则计算。
7. 运营 Agent 只能生成活动配置草案，不能直接应用活动。
8. 运营 Agent 生成的 tuning 字段必须经过人工确认和后续 schema 校验。
9. 配置工作台应用草案前必须通过 schema 校验。
10. 回滚操作必须保留历史记录，不能静默删除已应用配置。
