# 创新点记录

## 1. AI LiveOps 闭环雏形

要解决的问题：

传统小游戏 Demo 往往只有玩法，没有运营闭环，难以证明我理解游戏上线后的数据驱动工作。

传统做法：

玩家完成操作后只更新游戏状态，不沉淀可分析事件。

本项目做法：

每次补货、接待、成交、流失、收摊都会记录事件，运营看板和 AI 日报都从这些事件中读取证据。

技术实现：

前端维护 `GameEvent` 列表，并基于事件计算转化率、收入、流失和日维度表现。

可展示效果：

可以一边玩，一边切到运营页看到指标变化。

这是一个最小 LiveOps 闭环，后续可以自然扩展到后端埋点、A/B 测试和活动调参。


## 2. 受规则约束的模拟 Agent

要解决的问题：

LLM 如果直接修改游戏状态，容易产生不可控结果。

传统做法：

把 AI 对话结果直接作为游戏结果，缺少边界。

本项目做法：

第一版先用本地规则模拟 Agent 报告，明确“Agent 只能建议，不能直接改写金币或声望”。

技术实现：

Agent 报告读取游戏状态和指标，生成结构化活动草案，并在配置中写入 guardrail。

可展示效果：

AI 工作台展示证据、风险、建议和配置草案。

这个设计体现了 Agent 工程中的可控性和人审机制，后续可替换为真实模型调用。


## 3. 面向作品集展示的可观测 Agent 工作台

要解决的问题：

很多 AI 项目只展示最终回答，您看不出系统如何工作。

传统做法：

只显示一段 AI 文案。

本项目做法：

Agent 页同时展示运营日报、工具调用轨迹和 JSON 配置草案。

技术实现：

从事件数量、商品配置和报告生成状态中构造可观测轨迹。

可展示效果：

能看到 Agent 读取了哪些数据、生成了什么建议、哪些内容需要人工确认。

强调 AI Agent 不是“黑盒聊天”，而是可解释、可审计、可接入工具链的系统。


## 4. NPC 结构化记忆与规则决策分离

要解决的问题：

AI NPC 如果只有一次性对话，很难体现“角色会记住玩家”的游戏感；如果完全由模型决定结果，又容易破坏数值平衡。

传统做法：

NPC 每次出现都按固定台词或随机选择商品，不保留历史互动。

本项目做法：

每个 NPC 有独立记忆，包括好感、到访次数、最近购买、最近反馈和动态情绪。NPC Agent 读取这些记忆生成反馈，但购买、好感和数值变化仍由规则系统校验。

技术实现：

在前端状态中维护 `npcMemories` 和 `agentTraces`，接待时调用规则函数读取库存、价格、偏好、厌恶项和历史记忆，生成可解释决策。

可展示效果：

同一个 NPC 再次出现时，会根据上次购买和好感改变对话、情绪和接受替代商品的可能性。

强调“生成式体验”和“游戏规则稳定性”可以分层处理：Agent 负责解释、记忆和建议，规则系统负责最终状态变更。


## 5. NPC Agent 工具调用可视化

要解决的问题：

很难从一段 NPC 对话判断我是否真的做了 Agent 工具调用设计。

传统做法：

只显示 NPC 说了什么。

本项目做法：

Agent 页展示最近一次 NPC 决策的工具读取内容，包括库存、价格、NPC 记忆和任务状态，并展示规则边界。

技术实现：

每次接待都会生成 `AgentTrace`，记录工具读取、推荐决策和 guardrail。

可展示效果：

切到 Agent 页即可看到“为什么这个 NPC 买了这个商品”。

这能把 Agent 的推理输入、动作边界和最终决策变成可审计的工程产物。


## 6. 事件流驱动的运营诊断后台

要解决的问题：

很多游戏 Demo 只有实时状态，没有运营分析能力，无法证明我理解上线后的数据工作。

传统做法：

直接显示金币、收入等当前状态，缺少行为漏斗、任务完成和转化分析。

本项目做法：

所有运营指标都从事件流推导，包括任务完成率、商品购买转化、NPC 互动、流失节点和 A/B 版本对比。

技术实现：

在 `buildMetrics` 中基于 `GameEvent` 计算运营指标，运营页只消费结构化指标，不直接散落业务判断。

可展示效果：

玩家接待顾客后，运营后台会同步更新商品转化、NPC 互动和任务完成情况。

运营后台不是装饰页，而是围绕“埋点 - 指标 - 诊断 - 实验”的真实 LiveOps 工作流设计。


## 7. 面向小样本的 A/B 测试演示

要解决的问题：

早期作品集没有真实玩家量，很难展示运营实验思维。

传统做法：

等有大量数据后才做 A/B 测试，作品集阶段无法演示。

本项目做法：

先用当前事件流生成一个模拟对照版本，将“当前版本”和“夜灯补贴”版本放在同一运营面板中比较。

技术实现：

基于当前转化、流失和满意度计算两个版本的转化、客均收入和留存评分，并给出推荐版本。

可展示效果：

可以说明：当前是实验后台雏形，后续接入真实分流和实验 ID 即可落地。

体现我理解游戏运营中的实验设计、指标选择和阶段化落地。


## 8. AI 运营 Agent 的结构化日报

要解决的问题：

普通 AI 报告容易停留在泛泛建议，您难以判断它是否真的读取了运营数据。

传统做法：

让模型根据一段描述生成运营建议，输出不可追踪。

本项目做法：

Agent 读取事件流和运营指标，输出证据、风险、异常、原因假设、行动计划、调参建议、人工确认清单和活动配置草案。

技术实现：

`buildAgentReport` 消费 `buildMetrics` 的结构化指标，生成多段结构化对象，再由 Agent 工作台分模块展示。

可展示效果：

接待顾客后，Agent 页会根据实际库存、任务完成率、商品转化和 A/B 评分动态改变建议。

强调 AI Agent 是“读取数据 - 形成假设 - 输出动作 - 等待人审”的运营助手，而不是聊天框。


## 9. 人审优先的活动配置草案

要解决的问题：

AI 直接修改游戏配置存在幻觉、数值膨胀和不可回滚风险。

传统做法：

让 AI 输出一段自然语言活动方案，后续人工再手动整理。

本项目做法：

Agent 直接生成结构化配置草案，但标记 `approvalRequired` 和 guardrail，强调必须人工确认后才能进入游戏。

技术实现：

活动草案包含目标人群、触发条件、补贴、调参项、预期影响和安全边界。

可展示效果：

可以打开 Agent 页和配置页，说明建议如何从报告进入配置，但不会直接污染当前局内状态。

这体现了真实 Agent 工程里的安全边界、配置治理和运营协作流程。


## 10. 可回滚的配置热更新工作台

要解决的问题：

AI 生成运营方案后，如果无法进入配置系统，就只能停留在建议层；如果直接生效，又存在风险。

传统做法：

运营或策划手动复制 AI 建议，再改配置文件，缺少校验和回滚演示。

本项目做法：

配置工作台把 AI 活动草案接入 schema 校验、人工确认、应用、历史记录和回滚流程。

技术实现：

游戏状态维护 `activeConfig` 和 `configHistory`，应用配置会追加 `config_apply` 事件，回滚会追加 `config_rollback` 事件。

可展示效果：

点击“应用草案”后，首购补贴会真正影响局内金币；点击“回滚配置”后，当前配置恢复到上一版。

不仅会让 AI 生成内容，还能把内容纳入可控、可审计、可回滚的工程流程。


## 11. 配置影响玩法的最小闭环

要解决的问题：

很多配置页只展示 JSON，没有证明配置真的能改变游戏体验。

传统做法：

配置系统和玩法系统分离，Demo 阶段无法看到效果。

本项目做法：

应用 AI 配置后，首购补贴和偏好商品补货优先级会立即进入当前经营循环。

技术实现：

成交时读取 `first_purchase_subsidy`，补货时读取 `preferred_item_restock_priority`，由当前 `activeConfig` 控制结果。

可展示效果：

应用草案后第一次成交会显示“活动补贴 +3 铜钱”，并改变金币和收入。

这证明项目完成了“AI 建议 - 配置确认 - 游戏热更新 - 玩家体验变化”的 LiveOps 闭环。


## 12. Web 到桌面端的演进路径

要解决的问题：

作品集如果只停留在网页 Demo，容易被理解为一次性前端练习，而不是可演示、可交付的工具型产品。

传统做法：

先做完整桌面端，导致早期开发成本高，或者只做网页，缺少本地演示和打包能力。

本项目做法：

先用 Web 快速验证玩法、运营后台、AI Agent 和配置热更新，再用 Tauri 封装成桌面应用。

技术实现：

Tauri 复用 Vite 构建产物，桌面端通过统一脚本启动和打包，窗口尺寸按展示演示场景配置。

可展示效果：

同一套功能既能在浏览器中迭代，也能作为本地桌面应用打开和打包。

这体现了从原型到可交付应用的阶段化工程能力，也回应了游戏运营工具常见的本地演示、内网工具和桌面控制台场景。


## 13. 怪谈事件驱动的轻策略玩法

要解决的问题：

经营小游戏如果只有补货和接待，容易像数据表单，缺少“今天不一样”的游戏感。

传统做法：

用随机事件只改变文案，或者只做视觉装饰，不影响实际经营规则。

本项目做法：

新增怪谈事件卡，让每日事件影响预算、商品魅力、额外打赏和满意度。

技术实现：

游戏状态维护 `activeMysteryEvent`，接待顾客时读取事件并调整预算、商品选择、成交奖励和 Agent 工具轨迹。

可展示效果：

第 1 天“夜雾压街”会压低预算；第 2 天“纸灯将熄”会提高纸灯笼和醒魂茶魅力；第 3 天“神秘订单”会给高魅力商品额外打赏。

向您讲解时：

这说明玩法增强不是单纯加皮肤，而是把主题、规则、埋点和 Agent 输入连接起来。


## 14. 可替换的主题美术资产层

要解决的问题：

作品集早期如果完全依赖外部图片生成，容易因为素材没准备好而阻塞开发；如果完全没有图片，又缺少游戏主题表现力。

传统做法：

先做纯 UI 占位，等最后再补图，导致玩法和美术脱节。

本项目做法：

先用本地 SVG 建立稳定的图片资产层，包括背景、商品图标和 NPC 头像，后续可逐步替换为 image-2 精细位图。

技术实现：

商品、NPC 数据直接引用资产路径，UI 只关心 `icon` 和 `portrait` 字段，不关心具体图片来源。

可展示效果：

游戏主界面已经有怪谈夜市背景、商品图标、NPC 头像和事件卡，比纯后台式界面更像游戏。

向您讲解时：

这体现了“先做可运行资产管线，再逐步提升美术质量”的工程思路。


## 15. NPC 关系线转化为经营奖励

要解决的问题：

普通经营小游戏里的 NPC 往往只是购买者，玩家很难感到“我认识了这个角色”，运营后台也很难解释某条剧情线是否真的影响留存和经营决策。

传统做法：

把支线任务做成一次性对话或一次性奖励，完成后只给金币、道具或一段文本，和后续经营循环关系较弱。

本项目做法：

新增“灯纸匠的旧灯约”：玩家向灯纸匠成功卖出 2 次纸灯笼后，纸灯笼补货成本永久降低 2 铜钱。

技术实现：

`relationshipQuest` 保存任务状态，成交时读取 NPC、商品和任务状态推进进度；完成后 `getRestockCost` 会读取任务奖励并改变补货价格，同时写入 `relationship_quest` 事件。

可展示效果：

游戏页能看到关系任务进度和完成奖励，运营页能看到关系进度 KPI，Agent 页能看到 `read_relationship_quest` 工具轨迹和剧情运营行动建议。

这让剧情内容不只是装饰，而是进入了数值、埋点和 AI Agent 建议链路，适合作为游戏运营/AI Agent 岗位作品集中的创新点。


## 16. 成就系统作为早期留存反馈

要解决的问题：

轻量经营游戏如果只看金币、库存和满意度，玩家的短期行为很难形成明确成长感，运营侧也缺少判断“玩家是否理解目标”的中间指标。

传统做法：

成就只作为展示徽章，或者只在最终结算时统一弹出，难以及时影响玩家行为和运营分析。

本项目做法：

新增“夜市成就”系统，把首单成交、关系线完成、回头客信任和低流失经营做成可即时解锁的短期目标。

技术实现：

`evaluateAchievements` 在成交和收摊后统一判定成就，解锁后写入 `achievement` 事件；运营后台读取成就解锁率，Agent 报告读取成就证据并生成“成长反馈”诊断和行动计划。

可展示效果：

玩家接待第一位顾客后会看到“第一盏灯亮”解锁；无流失收摊后会看到“稳摊一夜”按真实经营日解锁；运营页和 Agent 页会同步更新。

向您讲解时：

这说明项目不只是堆玩法，而是把玩家成长、行为埋点、运营诊断和 Agent 建议连接起来，适合讲早期留存、任务引导和 LiveOps 指标设计。


## 17. 事件插图绑定规则配置

要解决的问题：

游戏美术如果只作为背景装饰，玩家可能看不出图片和当前规则有什么关系，运营侧也无法说明素材如何服务玩法变化。

传统做法：

事件只用文字说明，或者用固定装饰图，不随事件规则切换。

本项目做法：

为夜雾压街、纸灯将熄、神秘订单分别新增事件插图，并把插图挂到 `MysteryEvent` 配置中。

技术实现：

`MysteryEvent` 新增 `illustration` 字段，游戏事件卡读取 `activeMysteryEvent.illustration` 渲染图片；规则仍由预算、魅力和打赏函数读取同一个事件对象。

可展示效果：

第 1 天显示夜雾插图和预算变化，第 2 天显示灯火将熄插图和魅力变化，第 3 天显示神秘订单插图和打赏变化。

向您讲解时：

这能说明美术资产不是“贴图装饰”，而是和配置、事件、规则、玩家决策绑定的一部分，体现游戏运营工程中的内容可配置化思路。


## 18. 怪谈事件的玩家经营抉择

要解决的问题：

随机事件如果只自动改变数值，玩家会觉得自己只是被系统推着走，缺少“我选择了承担什么风险”的经营感。

传统做法：

事件触发后自动生效，例如今天减预算、明天加魅力，玩家只能被动适应。

本项目做法：

每个怪谈事件都提供两种经营抉择：夜雾可以花钱点灯或省灯油，灯火将熄可以换灯芯或借暗卖面，神秘订单可以接下或婉拒。

技术实现：

`eventChoices` 保存玩家选择，`event_choice` 写入事件流；预算、魅力、打赏和未成交惩罚函数读取当日抉择后再计算结果。

可展示效果：

玩家在接待前选择“点亮引路灯”，夜雾预算惩罚会失效；选择“接下神秘订单”，高魅力商品打赏提高，但未成交会额外损失满意度。

向您讲解时：

这可以讲清楚游戏运营里的风险决策设计：同一个事件既能驱动玩法分叉，也能形成可分析的玩家选择数据，供运营后台和 Agent 继续诊断。

## 19. 每日目标作为玩法与运营共同语言

要解决的问题：

轻量经营游戏如果只有自由点击，玩家容易不知道当天应该追求什么；运营后台也只能看到泛化指标，很难解释“目标设计是否有效”。

传统做法：

用一行任务文案提示玩家，或者只在最终结算时显示是否完成。这样目标和实际经营数据、Agent 建议之间连接较弱。

本项目做法：

将每日目标结构化为标题、描述、奖励和子条件，让玩家在游戏主界面看到进度，让运营后台看到目标结算，让 Agent 读取目标压力并提出奖励调优建议。

技术实现：

新增 `buildDailyGoal`、`getDailyGoalProgress`、`isDailyGoalCompleted` 统一计算每日目标。收摊时写入 `daily_goal` 事件；运营后台读取目标结算和子条件；Agent 报告读取当前目标、目标进度和达成次数。

可展示效果：

玩家能看到“雾夜开张”的接待和收入进度；收摊后事件流出现 `daily_goal`；运营页显示目标结算；Agent 页显示“每日目标压力”和“校准每日目标奖励”。

向您讲解时：

这能说明我理解游戏目标设计不只是 UI 文案，而是玩家行为牵引、运营指标和 Agent 调优输入之间的共同数据结构。


## 20. 声望分支驱动顾客与风险

要解决的问题：

声望如果只作为一个数字显示，玩家很难感受到它会改变经营局势，运营后台也很难解释“高声望”和“低声望”分别带来什么行为变化。

传统做法：

声望只影响最终评分，或者只在结算时给一段评价，不进入 NPC 到访、商品价值和风险惩罚。

本项目做法：

把声望拆成“巡夜调查 / 街坊观望 / 灯影贵客”三种分支。低声望带来预算压力和额外流失惩罚，高声望吸引稀有顾客并强化稀有商品魅力。

技术实现：

新增 `getReputationBranch` 派生分支状态，`getNpcForReputation` 影响下一位顾客，预算、魅力和未成交惩罚函数读取分支。运营后台展示声望分支 KPI，Agent 读取 `read_reputation_branch` 并生成诊断和行动计划。

可展示效果：

游戏页能看到“街坊观望”声望卡；高声望会更容易遇到狐面少年，狐面具魅力提升；低声望会触发“巡夜调查”风险。Agent 页可以解释声望分支如何影响当前推荐。

向您讲解时：

这能说明我把“数值成长”做成了可解释的系统分支，而不是单纯加一个分数；同时体现了玩法规则、事件埋点、运营看板和 Agent 工具输入的联动。


## 21. 商品组合把补货变成策略

要解决的问题：

经营小游戏如果每个商品只是独立的价格、成本和魅力，玩家最优策略会很快变成“补最赚钱的货”，游戏性和运营分析空间都偏窄。

传统做法：

商品只做单件售卖，或者用简单套装折扣提升客单价，但不和 NPC 偏好、事件、库存缺口、Agent 诊断连接。

本项目做法：

新增“暖灯茶点 / 巡夜口粮 / 灯下面具传闻”三组商品组合。只有成套商品都有库存时组合才激活，激活后会影响商品魅力、成交奖励、满意度、声望或收入。

技术实现：

新增 `productCombos` 配置、`getActiveProductCombos` 派生函数、组合魅力和成交奖励计算函数。接待顾客时推荐逻辑读取组合魅力，成交后写入 `product_combo` 事件。运营后台展示组合激活率和触发次数，Agent 工具轨迹读取 `read_product_combos`。

可展示效果：

开局 3 组组合全部激活；当玩家卖掉醒魂茶后，“暖灯茶点”会因缺货失效。运营页能看到组合激活率变化，Agent 页能解释当前应该补齐哪些成套商品。

向您讲解时：

这能说明我不仅会写功能，还会把“商品系统、库存策略、埋点分析、AI Agent 可解释建议”做成一个闭环，符合游戏运营和 Agent 工程岗位的交叉能力。


## 22. 确定性怪谈热度替代黑盒随机

要解决的问题：

需求中希望商品组合能影响风险和事件触发概率，但如果直接做随机，展示演示时很难稳定复现，运营后台也很难解释某个事件为什么出现。

传统做法：

用固定事件轮转，或者在收摊时随机抽取下一事件。前者缺少策略反馈，后者虽然有变化，但对玩家和 Agent 来说都不够可解释。

本项目做法：

新增“怪谈热度”评分。商品组合、稀有货库存、声望和满意度共同影响热度，热度再影响下一日事件倾向和稀有顾客到访。

技术实现：

`productCombos` 新增 `riskWeight` 和 `eventBias`。`getRumorForecast` 统一计算热度、事件评分和下一日倾向；收摊时写入 `rumor_heat` 事件；运营后台展示热度 KPI 和热度快照；Agent 工具轨迹读取 `read_rumor_forecast`。

可展示效果：

玩家能在游戏页看到“怪谈热度”和下一日倾向；收摊后消息会显示热度值和下一日事件；运营页能看到热度指标；配置页能看到事件评分输入。

向您讲解时：

这体现了游戏运营岗位需要的“可控随机”：既让游戏产生变化，又让变化可解释、可埋点、可调参、可复现，适合用来说明 Agent 工程中的可观测性和边界控制。


## 23. 展示演示用本地快照闭环

要解决的问题：

作品集项目如果刷新页面就丢进度，展示演示会很脆弱；如果只能口头讲数据闭环，您也不容易拿到真实事件流证据。

传统做法：

只把状态放在内存中，或者等后端/数据库完成后再考虑持久化。这样早期演示不稳定，也不利于展示“运营数据可复盘”。

本项目做法：

先用 `localStorage` 做 Web 和 Tauri 都可复用的自动存档，并提供 JSON 快照导出。快照包含游戏状态、事件流、关键指标、Agent 报告和配置草案。

技术实现：

新增 `SavedGamePayload`、`loadSavedGame`、`persistGameLocally` 和 `exportGameSnapshot`。React 状态初始化时优先读取本地存档，状态变化后自动写入；游戏页显示本地存档卡；配置页输出 `localPersistence`；Agent 建议导出演示数据快照。

可展示效果：

玩家接待顾客后刷新页面，进度仍然保留；展示演示时可以点击“导出演示快照”，得到一份可复盘 JSON。

向您讲解时：

这能说明我考虑了作品集交付稳定性、桌面端本地数据需求和运营复盘证据，而不是只做一次性前端状态 Demo。


## 24. 可复现局面的演示快照导入

要解决的问题：

作品集向您展示时，优秀局面如果只能靠现场一步步玩出来，容易受随机事件、时间和操作顺序影响；只导出数据也只能复盘，不能复现。

传统做法：

准备截图或录屏，或者现场从新局开始演示。这样很难展示 Agent 如何读取复杂局面并生成建议。

本项目做法：

在本地存档和 JSON 导出的基础上，增加“导入演示快照”。快照通过 schema、版本号和最小游戏状态校验后，可以直接恢复到指定经营日。

技术实现：

新增 `DemoSnapshotPayload`、`readImportedGameSnapshot` 和文件输入导入流程。导入成功后调用 `setGame` 恢复局面，并继续由自动存档写入 `localStorage`。Agent 工作台和配置视图同步暴露导入能力，说明这是桌面演示链路的一部分。

可展示效果：

展示前导出一份包含高热度、商品组合、关系任务和 Agent 报告的快照；现场导入后，页面立即回到该局面，刷新后也不会丢失。

向您讲解时：

这能说明我不只是做“能跑的 Demo”，而是考虑了展示、运营复盘和桌面工具交付里的可复现性。


## 25. 展示场景化多演示槽位

要解决的问题：

一个作品集 Demo 往往需要展示多个局面：初始经营、怪谈高热度、关系任务完成、配置回滚后效果。如果只有一个最新存档，现场切换成本很高。

传统做法：

用多个录屏或截图讲解，或者现场重玩到目标状态。前者缺少交互性，后者不稳定。

本项目做法：

在本地自动存档之外，新增 3 个本地演示槽位。玩家可以把当前局面保存成槽位，之后一键读取或删除。

技术实现：

新增 `SaveSlot`、`readSaveSlots`、`persistSaveSlots` 和 `createSaveSlot`。槽位独立存储在 `second-game.local-save-slots.v1`，读取后复用 `setGame` 和自动存档写回当前状态。

可展示效果：

展示前可以分别保存“高热度传闻”“关系任务完成”“配置补贴生效”等局面，现场按需要读取，马上展示对应的运营看板和 Agent 建议。

向您讲解时：

这体现了我对作品集交付的控制力：不仅功能能跑，还能稳定组织演示路径，让您看到多个系统能力。


## 26. 本地运营日志证据包

要解决的问题：

游戏运营/AI Agent 工程师作品集不能只展示“界面好看”，还需要证明 Agent 建议背后有事件流、指标、配置历史和工具轨迹支撑。

传统做法：

用截图或口头解释运营判断，缺少可下载、可复核的数据证据。您很难判断我是否真的理解 LiveOps 数据闭环。

本项目做法：

在运营诊断后台新增“运营日志导出”，把当前局面的摘要、指标、Agent 报告、事件流、工具轨迹和配置历史导出为本地 JSON。

技术实现：

新增 `exportOpsLog`，在 `OpsDashboard` 中接入 `report`、`metrics` 和 `game`，并在配置视图暴露 `localPersistence.opsLogExport`。Agent 行动计划和调参建议同步记录这个能力。

可展示效果：

向您展示时可以先操作局内经营，再打开运营页下载证据包，说明每条建议来自哪些埋点、异常假设和配置变更。

向您讲解时：

这体现了从游戏玩法到运营工具再到 Agent 复盘的完整工程思路：不是让 AI 凭空给建议，而是把可观察数据沉淀为可复核资产。


## 27. 本地演示重置与槽位隔离

要解决的问题：

作品集展示会反复演示同一个游戏。如果每次都要手动清浏览器缓存或重新导入快照，现场节奏容易被打断；但一键清空又可能误删提前准备好的高价值演示槽位。

传统做法：

依赖浏览器开发者工具清 localStorage，或者把“重新开始”只当成游戏内重开，不解释它如何影响本地存档。

本项目做法：

在本地存档卡中新增“重置本地演示”。它会回到干净 Day 1 并覆盖当前自动存档，但保留独立的演示槽位。

技术实现：

`resetGame` 复用 `makeInitialState` 和自动存档 effect，并通过 `localSaveNoticeRef` 给出“当前自动存档已覆盖、槽位仍保留”的反馈。配置视图新增 `localPersistence.resetCurrentDemo`，Agent 行动计划和调参建议同步说明该能力。

可展示效果：

向您展示时可以先展示复杂局面，再一键回到新摊开张；需要继续讲高热度或关系任务时，再从槽位读取预设局面。

向您讲解时：

这体现了面向演示和桌面交付的产品意识：本地数据不是简单“存或不存”，而是要分清当前自动存档、可复现快照和多组演示槽位的生命周期。


## 28. Tauri 原生菜单驱动的桌面演示闭环

要解决的问题：
作品集如果只有 Web 页面和桌面壳，您可能会觉得“桌面版只是套了一层窗口”。对于游戏运营/AI Agent 工程岗位，更有说服力的是：同一套玩法、数据、Agent 报告和本地演示能力，能被包装成一个具备桌面入口和交付边界的工具。

传统做法：
只保留页面内按钮，或者为了桌面端单独重写一套 UI。前者桌面属性弱，后者容易造成 Web 与桌面行为不一致，维护成本也高。

本项目做法：
Tauri 端新增“文件”“视图”“帮助”原生菜单。“视图”负责切换四个工作区；“文件”负责导出演示快照、重置本地演示、刷新和退出；“帮助”负责打开 GitHub 仓库。菜单不会直接改游戏状态，而是向 React 发出结构化事件。

技术实现：
Rust 侧使用 Tauri 2 的 `MenuBuilder` 和 `SubmenuBuilder` 创建菜单，并通过 `app.emit` 发出 `desktop-menu://tab` 与 `desktop-menu://action`。前端只在 Tauri 环境动态导入 `@tauri-apps/api/event` 并监听事件，复用已有 `setActiveTab`、`exportSnapshot` 和 `resetGame`。

可展示效果：
向您展示时可以从系统菜单切换游戏、运营后台、Agent 工作台和配置页，也可以直接导出或重置演示现场。这个动作能把“网页 Demo”升级成“可本地交付的运营工具”来讲。

向您讲解时：
这体现了我对交付形态的判断：桌面端不是另起炉灶，而是在稳定 Web 功能上增加原生入口、版本元数据和演示闭环，让项目更贴近内部运营工具、GM 工具和本地 AI Agent 控制台。
