# 创新点记录

## 1. AI LiveOps 闭环雏形

要解决的问题：

传统小游戏 Demo 往往只有玩法，没有运营闭环，难以证明候选人理解游戏上线后的数据驱动工作。

传统做法：

玩家完成操作后只更新游戏状态，不沉淀可分析事件。

本项目做法：

每次补货、接待、成交、流失、收摊都会记录事件，运营看板和 AI 日报都从这些事件中读取证据。

技术实现：

前端维护 `GameEvent` 列表，并基于事件计算转化率、收入、流失和日维度表现。

可展示效果：

面试时可以一边玩，一边切到运营页看到指标变化。

面试讲解角度：

这是一个最小 LiveOps 闭环，后续可以自然扩展到后端埋点、A/B 测试和活动调参。

## 2. 受规则约束的模拟 Agent

要解决的问题：

LLM 如果直接修改游戏状态，容易产生不可控结果。

传统做法：

把 AI 对话结果直接作为游戏结果，缺少边界。

本项目做法：

第一版先用本地规则模拟 Agent 报告，明确“Agent 只能建议，不能直接改写金币或声望”。

技术实现：

Agent 报告读取游戏状态和指标，生成结构化活动草案，并在配置中写入 guardrail。

可展示效果：

AI 工作台展示证据、风险、建议和配置草案。

面试讲解角度：

这个设计体现了 Agent 工程中的可控性和人审机制，后续可替换为真实模型调用。

## 3. 面向作品集展示的可观测 Agent 工作台

要解决的问题：

很多 AI 项目只展示最终回答，面试官看不出系统如何工作。

传统做法：

只显示一段 AI 文案。

本项目做法：

Agent 页同时展示运营日报、工具调用轨迹和 JSON 配置草案。

技术实现：

从事件数量、商品配置和报告生成状态中构造可观测轨迹。

可展示效果：

面试官能看到 Agent 读取了哪些数据、生成了什么建议、哪些内容需要人工确认。

面试讲解角度：

强调 AI Agent 不是“黑盒聊天”，而是可解释、可审计、可接入工具链的系统。

## 4. NPC 结构化记忆与规则决策分离

要解决的问题：

AI NPC 如果只有一次性对话，很难体现“角色会记住玩家”的游戏感；如果完全由模型决定结果，又容易破坏数值平衡。

传统做法：

NPC 每次出现都按固定台词或随机选择商品，不保留历史互动。

本项目做法：

每个 NPC 有独立记忆，包括好感、到访次数、最近购买、最近反馈和动态情绪。NPC Agent 读取这些记忆生成反馈，但购买、好感和数值变化仍由规则系统校验。

技术实现：

在前端状态中维护 `npcMemories` 和 `agentTraces`，接待时调用规则函数读取库存、价格、偏好、厌恶项和历史记忆，生成可解释决策。

可展示效果：

同一个 NPC 再次出现时，会根据上次购买和好感改变对话、情绪和接受替代商品的可能性。

面试讲解角度：

强调“生成式体验”和“游戏规则稳定性”可以分层处理：Agent 负责解释、记忆和建议，规则系统负责最终状态变更。

## 5. NPC Agent 工具调用可视化

要解决的问题：

面试官很难从一段 NPC 对话判断候选人是否真的做了 Agent 工具调用设计。

传统做法：

只显示 NPC 说了什么。

本项目做法：

Agent 页展示最近一次 NPC 决策的工具读取内容，包括库存、价格、NPC 记忆和任务状态，并展示规则边界。

技术实现：

每次接待都会生成 `AgentTrace`，记录工具读取、推荐决策和 guardrail。

可展示效果：

切到 Agent 页即可看到“为什么这个 NPC 买了这个商品”。

面试讲解角度：

这能把 Agent 的推理输入、动作边界和最终决策变成可审计的工程产物。

## 6. 事件流驱动的运营诊断后台

要解决的问题：

很多游戏 Demo 只有实时状态，没有运营分析能力，无法证明候选人理解上线后的数据工作。

传统做法：

直接显示金币、收入等当前状态，缺少行为漏斗、任务完成和转化分析。

本项目做法：

所有运营指标都从事件流推导，包括任务完成率、商品购买转化、NPC 互动、流失节点和 A/B 版本对比。

技术实现：

在 `buildMetrics` 中基于 `GameEvent` 计算运营指标，运营页只消费结构化指标，不直接散落业务判断。

可展示效果：

玩家接待顾客后，运营后台会同步更新商品转化、NPC 互动和任务完成情况。

面试讲解角度：

强调运营后台不是装饰页，而是围绕“埋点 - 指标 - 诊断 - 实验”的真实 LiveOps 工作流设计。

## 7. 面向小样本的 A/B 测试演示

要解决的问题：

早期作品集没有真实玩家量，很难展示运营实验思维。

传统做法：

等有大量数据后才做 A/B 测试，作品集阶段无法演示。

本项目做法：

先用当前事件流生成一个模拟对照版本，将“当前版本”和“夜灯补贴”版本放在同一运营面板中比较。

技术实现：

基于当前转化、流失和满意度计算两个版本的转化、客均收入和留存评分，并给出推荐版本。

可展示效果：

面试时可以说明：当前是实验后台雏形，后续接入真实分流和实验 ID 即可落地。

面试讲解角度：

体现候选人理解游戏运营中的实验设计、指标选择和阶段化落地。

## 8. AI 运营 Agent 的结构化日报

要解决的问题：

普通 AI 报告容易停留在泛泛建议，面试官难以判断它是否真的读取了运营数据。

传统做法：

让模型根据一段描述生成运营建议，输出不可追踪。

本项目做法：

Agent 读取事件流和运营指标，输出证据、风险、异常、原因假设、行动计划、调参建议、人工确认清单和活动配置草案。

技术实现：

`buildAgentReport` 消费 `buildMetrics` 的结构化指标，生成多段结构化对象，再由 Agent 工作台分模块展示。

可展示效果：

接待顾客后，Agent 页会根据实际库存、任务完成率、商品转化和 A/B 评分动态改变建议。

面试讲解角度：

强调 AI Agent 是“读取数据 - 形成假设 - 输出动作 - 等待人审”的运营助手，而不是聊天框。

## 9. 人审优先的活动配置草案

要解决的问题：

AI 直接修改游戏配置存在幻觉、数值膨胀和不可回滚风险。

传统做法：

让 AI 输出一段自然语言活动方案，后续人工再手动整理。

本项目做法：

Agent 直接生成结构化配置草案，但标记 `approvalRequired` 和 guardrail，强调必须人工确认后才能进入游戏。

技术实现：

活动草案包含目标人群、触发条件、补贴、调参项、预期影响和安全边界。

可展示效果：

面试时可以打开 Agent 页和配置页，说明建议如何从报告进入配置，但不会直接污染当前局内状态。

面试讲解角度：

这体现了真实 Agent 工程里的安全边界、配置治理和运营协作流程。

## 10. 可回滚的配置热更新工作台

要解决的问题：

AI 生成运营方案后，如果无法进入配置系统，就只能停留在建议层；如果直接生效，又存在风险。

传统做法：

运营或策划手动复制 AI 建议，再改配置文件，缺少校验和回滚演示。

本项目做法：

配置工作台把 AI 活动草案接入 schema 校验、人工确认、应用、历史记录和回滚流程。

技术实现：

游戏状态维护 `activeConfig` 和 `configHistory`，应用配置会追加 `config_apply` 事件，回滚会追加 `config_rollback` 事件。

可展示效果：

点击“应用草案”后，首购补贴会真正影响局内金币；点击“回滚配置”后，当前配置恢复到上一版。

面试讲解角度：

这能说明候选人不仅会让 AI 生成内容，还能把内容纳入可控、可审计、可回滚的工程流程。

## 11. 配置影响玩法的最小闭环

要解决的问题：

很多配置页只展示 JSON，没有证明配置真的能改变游戏体验。

传统做法：

配置系统和玩法系统分离，Demo 阶段无法看到效果。

本项目做法：

应用 AI 配置后，首购补贴和偏好商品补货优先级会立即进入当前经营循环。

技术实现：

成交时读取 `first_purchase_subsidy`，补货时读取 `preferred_item_restock_priority`，由当前 `activeConfig` 控制结果。

可展示效果：

应用草案后第一次成交会显示“活动补贴 +3 铜钱”，并改变金币和收入。

面试讲解角度：

这证明项目完成了“AI 建议 - 配置确认 - 游戏热更新 - 玩家体验变化”的 LiveOps 闭环。

## 12. Web 到桌面端的演进路径

要解决的问题：

作品集如果只停留在网页 Demo，容易被理解为一次性前端练习，而不是可演示、可交付的工具型产品。

传统做法：

先做完整桌面端，导致早期开发成本高，或者只做网页，缺少本地演示和打包能力。

本项目做法：

先用 Web 快速验证玩法、运营后台、AI Agent 和配置热更新，再用 Tauri 封装成桌面应用。

技术实现：

Tauri 复用 Vite 构建产物，桌面端通过统一脚本启动和打包，窗口尺寸按面试演示场景配置。

可展示效果：

同一套功能既能在浏览器中迭代，也能作为本地桌面应用打开和打包。

面试讲解角度：

这体现了从原型到可交付应用的阶段化工程能力，也回应了游戏运营工具常见的本地演示、内网工具和桌面控制台场景。
