# 对面试官的介绍

## 一句话介绍

我做了一个“夜市怪谈摊主”的小游戏，并围绕它搭建了运营数据后台和 AI LiveOps Agent。玩家行为会变成埋点数据，Agent 基于数据生成运营建议和活动配置，配置经过人工确认后可以热更新进游戏，并且支持回滚。

## 为什么这个项目适合游戏运营 / AI Agent 工程岗位

1. 有可玩的小游戏核心循环，说明我能把产品体验做出来。
2. 有事件埋点和运营指标，说明我理解游戏上线后的数据运营。
3. 有 NPC Agent 和运营 Agent，说明我能把 Agent 拆成输入、工具、输出和安全边界。
4. 有配置热更新和回滚，说明我关注 AI 输出进入生产系统时的可控性。
5. 有 Tauri 桌面端，说明我考虑了作品集交付和本地演示。

## 技术亮点

1. React + TypeScript 实现游戏、后台和工作台。
2. 事件流作为单一事实来源，运营指标和 Agent 报告都从事件中计算。
3. NPC Agent 读取库存、偏好、预算和记忆，输出推荐理由和工具轨迹。
4. AI 运营 Agent 输出结构化日报、异常假设、行动计划、调参建议和活动草案。
5. 配置工作台提供 schema 校验、人工确认、应用、历史追踪和回滚。
6. Tauri 2 封装桌面端，支持本地启动和 Windows 安装包构建。

## 可以主动展开的问题

### 1. 为什么先用规则模拟 Agent？

因为早期目标是验证工程闭环，而不是追求模型效果。规则模拟可以先固定输入、输出和安全边界，后续接入真实模型时只需要替换生成层。

### 2. 如何避免 AI 幻觉影响游戏？

Agent 不能直接修改金币、声望、库存等关键状态。它只能生成建议和配置草案，配置进入游戏前必须通过 schema 校验和人工确认。

### 3. 这个项目如何扩展到真实生产？

可以把事件流接入后端数据库，把配置历史迁移到 SQLite 或服务端，增加真实 A/B 实验分流，再接入 OpenAI API 或兼容协议模型生成运营报告。

### 4. 为什么选择 Tauri？

Tauri 可以复用 Web 前端，桌面端体积更轻，也适合做运营工具、内部工具和本地演示版本。

## 30 秒收尾

这个项目的重点不是单点功能，而是闭环：小游戏产生数据，数据驱动 Agent，Agent 生成配置，配置经过治理后影响游戏体验。我希望用它证明自己同时理解游戏运营、前端工程和 AI Agent 落地中的可控性问题。
