# AI LiveOps 游戏运营 Agent 平台需求文档

## 1. 项目定位

项目暂定为一款轻量经营 + 剧情小游戏，例如《夜市怪谈摊主》。玩家经营夜市摊位，接待不同 NPC，管理库存、定价、任务与剧情选择。AI Agent 参与 NPC 行为、运营分析、活动策划、数值调参和配置生成，形成“游戏体验 - 数据采集 - AI 分析 - 运营配置 - 游戏迭代”的完整闭环。

## 2. 核心目标

1. 做出一个可玩的小游戏原型。
2. 建立游戏事件埋点和运营数据看板。
3. 构建 AI Agent，让它能够读取数据、调用工具、生成运营建议。
4. 支持运营活动、任务、数值配置的生成与热更新。
5. 从网页版逐步演进到桌面应用，桌面端优先考虑 Tauri。
6. 使用 Git 和 GitHub 做版本管理，开发过程需要可追踪；每一版开发完成后都必须上传到 GitHub。
7. 在整个开发过程中持续记录创新点、设计取舍和阶段复盘。
8. 后续增强版本需要提升游戏性和主题表现力，让项目从“可演示工具”进一步变成“更像游戏的作品”。

## 3. 目标用户

### 3.1 玩家

玩家希望在短时间内体验一个有反馈、有变化、有轻度剧情的经营小游戏。

玩家增强目标：

1. 每个游戏日不只是点击经营，而要有选择、风险、奖励和短期目标。
2. NPC 到访、事件触发、商品组合和剧情分支需要带来变化。
3. 视觉上要让玩家一眼感受到“怪谈夜市”的主题，而不是通用后台界面。

### 3.2 游戏运营 / 策划视角

运营人员希望看到玩家数据、流失节点、活动效果和可执行的调参建议。

## 4. 产品形态

### 4.1 Web 版本

第一阶段优先开发网页版，便于快速迭代、部署、展示和面试演示。

建议形态：

1. 游戏主界面。
2. 运营后台界面。
3. AI Agent 工作台。
4. 配置管理页面。

### 4.2 桌面版本

在 Web 版本稳定后，使用 Tauri 封装桌面应用。

桌面端目标：

1. 保留 Web 端主要功能。
2. 支持本地数据存储或本地缓存。
3. 支持一键启动游戏和运营后台。
4. 形成可安装、可演示的作品集应用。

## 5. 阶段规划

### 阶段 0：项目初始化

目标：

1. 建立项目目录结构。
2. 初始化 Git 仓库。
3. 绑定 GitHub 远程仓库并上传。
4. 建立开发日志和需求文档。
5. 确定技术栈和最小可行版本范围。

验收标准：

1. 存在 `docs/RequirementsDoc.md`。
2. 存在开发日志文档。
3. Git 仓库可以正常提交。
4. GitHub 仓库可以访问并持续同步，仓库地址为 `https://github.com/Ey1afjalla/second-game.git`。

备注：

1. GitHub 上传需要使用本机已有登录状态、GitHub CLI、SSH Key 或 Personal Access Token。
2. 不应把任何账号密码、API Key、Token 写入仓库。
3. 每一版完成后必须执行一次提交和推送，保证 GitHub 上的代码与本地阶段成果一致。

### 阶段 1：可玩小游戏 MVP

目标：

1. 实现基础经营循环：进货、摆摊、接待 NPC、销售、结算。
2. 实现基础数值：金币、库存、声望、天数、满意度。
3. 实现基础事件：随机顾客、特殊顾客、突发事件。
4. 实现基础 UI：游戏主屏、经营面板、对话面板、结算面板。

验收标准：

1. 玩家可以完成至少 3 个游戏日。
2. 每个游戏日有明确目标和结算结果。
3. 游戏数据可以被记录到本地或后端。

### 阶段 2：AI NPC Agent

目标：

1. 为 NPC 建立角色设定、偏好、预算、情绪和记忆。
2. 让 NPC 能根据玩家行为生成对话和反馈。
3. Agent 可以调用工具读取库存、价格、历史交互和任务状态。
4. AI 输出必须经过规则校验，不允许直接改写关键数值。

验收标准：

1. NPC 能体现不同性格和消费偏好。
2. NPC 能记住至少一次历史互动。
3. 对话结果能影响好感度、购买行为或任务进度。
4. 关键游戏状态由规则系统确认，而不是完全交给模型自由生成。

### 阶段 3：运营数据后台

目标：

1. 建立事件埋点系统。
2. 记录玩家行为、关卡进度、任务完成率、购买转化、流失节点等数据。
3. 建立运营看板，展示核心指标。
4. 支持基础 A/B 测试。

验收标准：

1. 可以查看玩家每个游戏日的行为数据。
2. 可以查看任务完成率、NPC 互动率、商品购买率。
3. 可以比较两个活动或数值版本的表现。

### 阶段 4：AI 运营 Agent

目标：

1. AI Agent 可以读取运营数据。
2. AI Agent 可以生成运营日报。
3. AI Agent 可以发现异常指标并提出原因假设。
4. AI Agent 可以生成活动方案和数值调整建议。

验收标准：

1. Agent 能生成包含问题、证据、建议和风险的运营报告。
2. Agent 能基于目标生成活动配置草案。
3. 运营建议能被人工确认后写入配置。

### 阶段 5：配置生成与热更新

目标：

1. 任务、商品、活动、NPC、奖励等内容通过配置驱动。
2. AI Agent 可以生成结构化 JSON 配置。
3. 配置进入游戏前需要通过 schema 校验。
4. 支持回滚到历史配置。

验收标准：

1. 修改配置后不需要改动核心代码即可改变游戏内容。
2. 错误配置会被拦截并给出提示。
3. 历史配置可以追踪和回滚。

### 阶段 6：Tauri 桌面应用

目标：

1. 将 Web 版本封装为 Tauri 桌面应用。
2. 支持本地运行、演示和打包。
3. 整合游戏、运营后台和 AI 工作台。

验收标准：

1. 可以启动桌面应用。
2. 核心功能与 Web 版一致。
3. 能形成可展示的安装包或可执行程序。

### 阶段 7：作品集打磨

目标：

1. 完善 README。
2. 编写架构说明。
3. 编写演示脚本。
4. 准备面试讲解材料。
5. 录制或截图展示关键功能。

### 阶段 8：游戏性与主题美术增强

目标：

1. 增加更明确的游戏目标、风险决策和成长反馈。
2. 增加夜市主题图片素材，包括背景、NPC 头像、商品图标和事件插图。
3. 让游戏主界面更符合“轻量经营 + 怪谈夜市”主题。
4. 保持运营后台和 Agent 工作台清晰专业，但游戏区可以更有氛围。
5. 让图片素材与配置、NPC、事件系统关联，而不是只作为装饰。

玩法增强建议：

1. 每日经营目标：例如当天必须赚够一定铜钱、接待指定 NPC、卖出指定商品。
2. 商品组合策略：不同商品有利润、稀有度、风险、NPC 偏好和事件触发概率。
3. 怪谈事件卡：夜雾、纸灯熄灭、无脸客、旧账本、雨夜涨价等事件改变当天规则。
4. NPC 关系线：多次接待同一 NPC 后解锁特殊对话、折扣、隐藏商品或任务。
5. 声望分支：高声望吸引稀有顾客，低声望触发怪谈风险和调查事件。
6. 经营抉择：是否高价卖给急需 NPC、是否保留稀有货、是否接受神秘订单。
7. 小型成就：例如“连续三天无流失”“让挑剔 NPC 回头”“触发隐藏商品传闻”。

验收标准：

1. 游戏主界面至少使用 1 张主题背景图、3 个商品图标和 3 个 NPC 头像。
2. 至少新增 3 种会改变规则或结果的怪谈事件。
3. 至少新增 1 条 NPC 关系线或轻剧情任务。
4. 图片素材风格统一，文件命名、用途和提示词有记录。
5. UI 自查确认图片不会压住文字，移动端和桌面端都能正常显示。
6. 新增玩法仍然能被事件埋点、运营后台和 AI Agent 读取。

## 6. 技术方案建议

### 6.1 前端

候选方案：

1. React + TypeScript。
2. Phaser 或 PixiJS 实现小游戏画面。
3. ECharts / Recharts 实现数据看板。
4. Tailwind CSS 或轻量设计系统实现 UI。

### 6.2 后端

候选方案：

1. Python FastAPI。
2. Node.js / NestJS。
3. SQLite 作为早期本地数据库。
4. PostgreSQL 作为后续扩展数据库。

### 6.3 AI Agent

候选方案：

1. OpenAI API / 兼容 OpenAI 协议的模型服务。
2. LangGraph 或自研轻量 Agent 编排。
3. 工具调用：读取玩家数据、读取配置、生成报告、生成配置草案。
4. 记忆系统：短期会话记忆 + 结构化玩家 / NPC 记忆。

### 6.4 桌面端

候选方案：

1. Tauri。
2. 前端复用 Web 项目。
3. 本地数据通过 SQLite 或文件存储。

### 6.6 UI 设计

如果条件允许，可以使用 Pencil AI 设计软件辅助 UI 设计。

UI 要求：

1. 优先做真实可用界面，而不是营销式首页。
2. 游戏主界面要有明确的经营信息层级。
3. 运营后台要安静、清晰、适合扫描数据。
4. AI 工作台要展示 Agent 的输入、推理依据、工具调用结果和输出。
5. 每个阶段都需要进行 UI 自查，避免文字溢出、布局错位、视觉风格割裂。

主题 UI 要求：

1. 游戏主界面可以使用背景插画、NPC 头像和商品图标增强沉浸感。
2. 信息面板应像“摊主账本”或“夜市木牌”，但文字必须保持高对比度。
3. 按钮可以使用铜钱、灯笼、符纸等视觉语言，但交互仍要清晰。
4. 运营后台和 Agent 工作台不要过度装饰，保留专业工具感，只用少量主题色统一品牌。
5. 游戏区、后台区和 Agent 区需要在风格上统一，但信息密度可以不同。
6. 图片只能辅助主题，不能遮挡核心数值、按钮、NPC 对话和配置内容。

当前候选创新点：

### 8.1 AI LiveOps 闭环

将小游戏玩家行为数据交给 AI Agent 分析，由 Agent 生成运营建议和活动配置，再由人工确认后影响下一轮游戏体验。

### 8.2 受规则约束的 AI NPC

NPC 对话由 AI 生成，但关键数值和状态变化由游戏规则系统校验，兼顾生成式体验和系统稳定性。

### 8.3 可回滚的 AI 配置生成

AI 可以生成任务、商品、奖励和活动配置，但所有配置必须经过 schema 校验、人工确认和版本记录。

### 8.4 面向面试展示的 Agent 可观测性

AI 工作台展示 Agent 调用了哪些工具、读取了哪些数据、为什么给出建议，让系统能力可解释、可演示。

### 8.5 运营目标驱动的活动生成

不是让 AI 随机生成活动，而是输入运营目标，例如“提高第 3 天留存”或“提升低活跃玩家回流”，Agent 根据数据和目标生成活动方案。

## 9. 版本管理要求

1. 使用 Git 管理全部代码和文档。
2. 使用 GitHub 作为远程仓库，远程仓库地址固定为 `https://github.com/Ey1afjalla/second-game.git`。
3. 每个阶段建立清晰提交记录。
4. 每一版完成后都必须推送到 GitHub，不能只保留在本地。
5. 重要节点打 tag，例如 `v0.1.0-web-mvp`、`v0.2.0-agent-npc`、`v1.0.0-tauri-demo`。
6. 每个 tag 创建后也需要推送到 GitHub。
7. 不提交账号密码、API Key、Token、私密配置。
8. 使用 `.env.example` 记录需要的环境变量名称。

## 10. 安全与隐私要求

1. 用户账号、密码、API Key、Token 只能保存在本地私密配置中。
2. 仓库中只保留 `.env.example`，不保留真实 `.env`。
3. AI 生成内容进入游戏前需要经过安全过滤和规则校验。
4. 玩家行为日志不记录真实个人隐私。
5. 外部模型调用需要记录成本、延迟和失败处理策略。

## 11. UI 自查清单

每次完成 UI 功能后需要检查：

1. 桌面宽屏是否布局正常。
2. 窄屏或小窗口是否可用。
3. 文字是否溢出。
4. 按钮、图标、表格和面板是否对齐。
5. 游戏界面和运营后台是否风格统一但信息层级明确。
6. AI 输出是否可读、可复制、可追踪。
7. 关键操作是否有加载、成功、失败和空状态。

## 12. 初始 MVP 范围

第一版建议只做以下内容：

1. 一个可玩的夜市经营循环。
2. 三类 NPC。
3. 五种商品。
4. 三个游戏日。
5. 基础事件埋点。
6. 一个运营看板。
7. 一个 AI 运营日报生成器。

暂不做：

1. 复杂战斗系统。
2. 大规模地图。
3. 多人在线。
4. 复杂付费系统。
5. 大量剧情分支。

## 13. 成功标准

项目完成后，应能支持以下表达：

1. “我做了一个完整的小游戏核心循环。”
2. “我设计了玩家行为埋点和运营指标。”
3. “我让 AI Agent 读取真实数据并生成运营建议。”
4. “我让 AI 生成的配置经过校验和人工确认后进入游戏。”
5. “我考虑了 AI 幻觉、成本、延迟、回滚和可观测性。”
6. “我从 Web 版本演进到了 Tauri 桌面应用，并使用 GitHub 进行版本管理。”
